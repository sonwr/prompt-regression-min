# Reviewer queue priority-rank hold vs post

Use this note after the CLI exposes `priority_rank` and before you post a human-facing reviewer handoff.

## Post it now when

- the winner has a clear rank lead,
- the next action is already known,
- the runner-up context can fit in one sentence.

## Hold it for another review pass when

- rank ties still dominate the queue,
- the lead only exists because of missing labels or incomplete grouping,
- the sentence would hide an important runner-up or source-case caveat.

## Safe one-line pattern

`Next reviewer focus: <winner> (rank <n>); runner-up: <label>; review the winner first unless tie context changes after the next run.`
