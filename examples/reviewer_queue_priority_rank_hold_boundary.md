# Reviewer queue priority-rank hold boundary

Use this note when the exposed priority rank is technically available, but the lead is too narrow to post as a decisive handoff.

## Hold conditions

Pause the public handoff when any of these are true:

- the winner and runner-up share the same priority rank,
- the queue-share lead is present but minimal,
- source-case impact is still effectively tied,
- or the owner action would be identical for both queues.

## Copy-ready note

```text
Priority rank is exposed, but the lead is still hold-sized. Keep both queues visible until the next batch or owner signal creates a cleaner separation.
```

## Escalate to a winner only when

- the winner has a distinct priority rank, or
- the winner keeps the same rank and also has a meaningful queue-share or source-case edge.

## Why this exists

A deterministic output should stay deterministic in wording too. If the data only supports a narrow hold, the handoff should say hold instead of pretending the winner is obvious.
